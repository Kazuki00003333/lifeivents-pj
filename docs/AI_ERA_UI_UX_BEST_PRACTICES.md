# AI時代のUI/UXベストプラクティス

**目的**: AIを組み込んだプロダクトにおいて、信頼・使いやすさ・コントロールを両立するUI/UXの原則と、LifePathへの適用の pointer をまとめる。

---

## 1. パラダイムの変化（チャットからタスクへ）

- **チャット中心からタスク・アウトカム中心へ**: AIのUIは「会話の往復」より「やることを決めて、結果を得る」設計が主流になりつつある。スライダー・ボタン・プリセット・テンプレで**入力を制御**し、**単一または少数の提案**を出してユーザーが選ぶ・編集する形が増えている。
- **人間の主体性（Human Agency）**: 「AIができるか」より「**AIの振る舞いが人の判断・主体性にどう影響するか**」が設計の中心になる。認知的主権・意味のある選択・説明責任を保つことが重要。
- **LifePathでの位置づけ**: チャット型AIではなく、**タスク提案・テンプレ生成・提携の「おすすめ」**など、**タスク・アウトカム型**のAIに寄せる。会話は補助（自然言語でイベント追加など）にとどめる。

---

## 2. 中核原則

### 2.1 透明性（Transparency）

| 観点 | ベストプラクティス | LifePathでの例 |
|------|-------------------|-----------------|
| **何ができるか** | AIの能力と**限界**を最初やヘルプで明示する | 「タスクの提案は目安です。追加・編集・削除は自由にできます」 |
| **なぜそうなったか** | 判断理由を短く説明できるようにする（Explainability） | 「七五三は3歳前後で行うことが多いため提案しました」のような一言 |
| **何をしているか** | 処理中は「AIがタスクを考えています」等、状態を表示する | ローディング＋短いメッセージ |
| **確信度** | 不確実な出力には信頼度を示す（「おすすめ」「目安」等のラベル） | 提案タスクに「推奨」バッジ、提携に「このイベントに合いやすい」等 |

規制（EU AI Act 等）を意識する場合は、AIであることのラベル・説明用のポップオーバーを用意する。

### 2.2 コントロール（Control）と「人間が最終決定」

| 観点 | ベストプラクティス | LifePathでの例 |
|------|-------------------|-----------------|
| **Governor パターン** | AIは提案し、**実行・確定はユーザーが明示的に行う** | タスク提案は「追加する」で確定。自動でイベントに追加しない |
| **Human-in-the-Loop** | 提案 → 確認・編集・却下 → 採用の流れを欠かさない | 提案タスク一覧で「全部追加」「選んで追加」「編集して追加」 |
| **取り消し** | 誤操作・不要なAI提案を**元に戻せる**ようにする | 追加したタスクの削除、または「提案を元に戻す」 |
| **フォールバック** | AIが失敗・不安定なときの**非AIの逃げ道**を用意する | 「テンプレから選ぶ」「手動でタスクを追加」 |

重要な決定（日付・予算・誰のイベントか）は**必ずユーザーが確認・編集できる**形にする。

### 2.3 出力の見せ方（Smart Output）

| タスクの性質 | 推奨 | 例 |
|--------------|------|-----|
| **決定的**（日付計算・形式変換など） | 1つの答えをはっきり表示 | 「本番の6ヶ月前 = 2025/12/15」 |
| **創作的・複数解あり**（タスク案・提携のおすすめ） | **1つを主提案**にしつつ、他を「もっと見る」で表示可能に | 「まずこの5件を追加しますか？」＋「他の提案を見る」 |
| **不確実** | 「目安」「参考」等のラベルと、編集前提の表示 | 「だいたいこの時期にやることが多いです」 |

プロンプト・プリセット・デフォルトで**ばらつきを抑え**、同じ条件なら同じような提案になるようにする。

### 2.4 エラー・制限・プライバシー

- **エラー時**: 「うまくいきませんでした。もう一度か、手動で追加してください」のように、**次のアクション**を明示する。
- **利用上限**: AIの利用回数に上限がある場合は、**事前に「あと○回」**を表示し、超過時は理由と**明日または課金で増える**ことを伝える（[PMF_INFRA_AND_AI_LIMITS.md](./PMF_INFRA_AND_AI_LIMITS.md) と整合）。
- **プライバシー**: イベント名・日付・家族情報などがAIに渡る場合は、設定や初回で**短く説明**し、オプトアウトや「AIを使わないで手動だけ」の選択肢を残す。

## 3. インタラクションのライフサイクル（Agentic AI のパターン）

アクションの**前・中・後**で次のパターンを揃えると、コントロールと説明責任が取りやすい。

| タイミング | パターン | 内容 |
|------------|----------|------|
| **実行前** | Intent Preview | 「この内容でタスクを5件追加します」と事前に要約表示 |
| **実行前** | Autonomy Dial | 自動度を選べる（「提案だけ見る」「選んで追加」「全部追加」など） |
| **実行中** | Explainable Rationale | 「七五三向けの一般的な準備を基準にしました」等の短い理由 |
| **実行中** | Confidence Signal | 「おすすめ」「目安」等のラベルやアイコン |
| **実行後** | Action Audit & Undo | 「追加したタスク」一覧と「元に戻す」「個別削除」 |
| **実行後** | Escalation Pathway | 「合わない・不具合」時のフィードバック・問い合わせ導線 |

LifePathでは、**タスク提案**と**提携のおすすめ**の両方に「実行前の確認」「実行後の取り消し」を必ず用意する。

## 4. レイアウト・配置のパターン

- **センターステージ**: メインコンテンツの中央にAIアシスタント（例: 自然言語入力）
- **右パネル**: 詳細・提案リスト・設定を右に（Web/タブレット）
- **インライン**: 入力欄の下やカードの横に「AI提案」を小さく表示
- **オーバーレイ**: モーダルで「提案を確認」してから確定

スマホメインなら**インライン**（タスク一覧の上に「AIで提案を追加」）と**オーバーレイ**（提案一覧 → 選択 → 追加）の組み合わせが扱いやすい。チャットを前面にしない。

---

## 5. 測定・検証

- **タスク完了率**: AI提案を使って「イベントの準備が完了した」まで到達した割合
- **意思決定時間**: 提案を見てから「追加する／編集する／却下する」までにかかる時間
- **満足度**: 短いアンケートやNPSで「AI提案は役に立ったか」を計測
- **誤り・訂正**: ユーザーが提案を削除・編集した頻度（改善のシグナル）

まず**価値の高いユースケース**（例: 七五三のタスク提案、葬儀チェックリスト）に絞り、速度・コスト・精度をチューニングしてから横展開する。

---

## 6. LifePath への適用チェックリスト

- [ ] AIの**能力と限界**をオンボーディングまたはヘルプで一言説明している
- [ ] タスク提案は**「追加する」で確定**し、自動で確定しない（Governor）
- [ ] 提案に**「おすすめ」「目安」**等のラベルを付け、編集前提であることを示す
- [ ] **取り消し・削除**がいつでもできる
- [ ] AI利用**回数と上限**を「あと○回」で表示し、超過時は理由と次の手段を表示
- [ ] 処理中は**「AIが考えています」**等の状態表示
- [ ] （任意）**提案理由**を短く表示（「七五三の一般的な準備から」等）
- [ ] テンプレ・手動追加など**AIなしのフォールバック**を用意している
- [ ] イベント・家族データをAIに送る場合は**説明と選択**を用意している

## 7. 参考

- Microsoft **Guidelines for Human-AI Interaction**（G1: 能力と限界の明示、G9: 効率的な修正・回復など）
- **Design Patterns for AI Interfaces**（Smashing Magazine 等）：タスク型UI、透明性、コントロール
- **Agentic AI** の UX パターン：Intent Preview, Autonomy Dial, Explainable Rationale, Action Audit & Undo, Escalation

---

**まとめ**: AI時代のUI/UXは**透明性・コントロール・人間が最終決定**を軸にし、チャットより**タスク・アウトカム型**に寄せる。LifePathでは提案→確認→追加・編集・却下の流れを明確にし、制限とフォールバックを必ず用意する。
